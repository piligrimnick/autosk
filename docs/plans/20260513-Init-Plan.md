# autosk v0.1 — Init Plan

**Date:** 2026-05-13
**Status:** Spec locked, ready for scaffold
**Scope:** MVP only. Out-of-scope items listed in §8.

---

## 1. Purpose

`autosk` is a minimal task tracker for AI coding agents. It mirrors the CLI
ergonomics of [beads](https://github.com/steveyegge/beads) but ships a tiny,
extensible surface area:

- One task table, four statuses, one edge kind.
- One storage backend now (doltlite, embedded, single-writer).
- One storage backend later (dolt server, multi-writer collaboration) behind
  the same Go interface.

The MVP is intentionally smaller than beads. Comments, workflows, labels,
hooks, federation, and search are deferred.

## 3. Data model

### 3.1 Task

```
Task
  id          string   "as-XXXX", primary key
  title       string   non-empty
  description string   may be empty
  status      enum     new | claimed | done | cancelled    (any → any)
  priority    int      0..3   (0 = highest, default = 2)
  created_at  int64    unix epoch seconds
  updated_at  int64    unix epoch seconds
```

Notes:
- `blocked` is **not** a stored status. A task is *blocked* iff it has at
  least one open blocker edge whose blocker is in `{new, claimed}`. This is
  computed at read time.
- No `assignee` / `actor` column in MVP. `claim` is anonymous.
- No `metadata` JSON column in MVP. Future fields arrive via migrations.

### 3.2 Dependency edge

```
Dep
  blocker_id   string   blocks blocked_id
  blocked_id   string
  kind         enum     'blocks'   (only value in MVP; column reserved for future kinds)
```

Invariants enforced by the schema:
- `blocker_id <> blocked_id` (no self-block)
- `(blocker_id, blocked_id, kind)` is unique
- `ON DELETE CASCADE` on both sides (a deleted task drops its edges)

Cycle prevention is enforced at the store layer (the `Block` operation
rejects edges that would close a cycle) — not in the schema.

### 3.3 Computed views

- **`blocked` flag** (per task): `EXISTS open blocker edge where blocker.status IN ('new','claimed')`.
- **Ready set**:
  ```sql
  status = 'new'
    AND NOT EXISTS (open blocker edge where blocker.status IN ('new','claimed'))
  ORDER BY priority ASC, created_at ASC
  ```
  Cancelled and done blockers do not block.

## 4. Command surface

14 commands. Every read command supports `--json`. Every write command
produces exactly one doltlite commit (so `autosk history` can recover field
history later).

### 4.1 Lifecycle

| Command | Behavior |
|---|---|
| `autosk create [title] [-d desc \| -d -] [-p N] [--blocks ID]... [--blocked-by ID]...` | Create a task. Title positional or `--title`. `-d -` reads description from STDIN. `--blocks` / `--blocked-by` add edges atomically with the create. |
| `autosk show <id>` | Pretty-print task + derived `blocked` flag + `blocked_by` and `blocks` id lists. |
| `autosk update <id> [--title S] [--description S] [--status S] [--priority N]` | Single SQL UPDATE. Validates status enum and priority range. |
| `autosk claim <id>` | Idempotent: `new \| claimed → claimed`. Returns `ErrNotClaimable` for `done \| cancelled`. |
| `autosk done <id>` | Set status `done`. |
| `autosk cancel <id>` | Set status `cancelled`. |
| `autosk reopen <id>` | `done \| cancelled → new`. |

### 4.2 Blocking (relationship-first)

| Command | Behavior |
|---|---|
| `autosk block <id> <blocker-id>...` | Variadic. Adds edges `blocker→id` for each. Single transaction. No status side-effect. |
| `autosk unblock <id> <blocker-id>... \| --all` | Variadic remove, or `--all` clears every incoming edge. Cobra enforces "ids OR --all". |
| `autosk dep list <id>` | Viewer only. Lists incoming (blocked_by) and outgoing (blocks) edges. |

`dep add` and `dep rm` are intentionally **not** present — `block` / `unblock`
are the only edge mutators.

### 4.3 Query

| Command | Behavior |
|---|---|
| `autosk list [--status S,S] [--priority N] [--limit N]` | Default filter: `status NOT IN (done, cancelled)`. Sort: priority ASC, created_at ASC. |
| `autosk ready [--limit N]` | Returns the ready set (see §3.3). |
| `autosk next` | Alias for `ready --limit 1`. |

### 4.4 Admin

| Command | Behavior |
|---|---|
| `autosk init [--prefix P]` | Explicit init. Optional — write commands auto-init. `--prefix` overrides the `as-` ID prefix (default `as`). |
| `autosk migrate` | Apply pending schema migrations. Also runs implicitly on every `Open`. |
| `autosk sql <query> [--write]` | Raw SQL passthrough. Refuses writes unless `--write`. |
| `autosk version` | Prints binary version + backend name + schema version. |
| `autosk history <id>` | **Stub in MVP.** Prints `"history delegates to doltlite log; planned for v0.2"`. Reserved so the command name stays free. |

## 5. Schema

Initial migration: `internal/migrations/001_init.sql`.

```sql
CREATE TABLE tasks (
  id          TEXT PRIMARY KEY,
  title       TEXT NOT NULL,
  description TEXT NOT NULL DEFAULT '',
  status      TEXT NOT NULL DEFAULT 'new'
              CHECK (status IN ('new','claimed','done','cancelled')),
  priority    INTEGER NOT NULL DEFAULT 2
              CHECK (priority BETWEEN 0 AND 3),
  created_at  INTEGER NOT NULL,
  updated_at  INTEGER NOT NULL
);
CREATE INDEX idx_tasks_status_prio ON tasks(status, priority, created_at);

CREATE TABLE task_deps (
  blocker_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  blocked_id TEXT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE,
  kind       TEXT NOT NULL DEFAULT 'blocks' CHECK (kind = 'blocks'),
  PRIMARY KEY (blocker_id, blocked_id, kind),
  CHECK (blocker_id <> blocked_id)
);
CREATE INDEX idx_deps_blocked ON task_deps(blocked_id);

CREATE TABLE schema_migrations (
  version    INTEGER PRIMARY KEY,
  applied_at INTEGER NOT NULL
);
```

Migrations are applied in numerical order and recorded in `schema_migrations`.
Each migration file is one transaction.

## 6. Storage interface

Single `Store` interface, two implementations from day one:

- `internal/store/doltlite/` — real MVP impl.
- `internal/store/doltserver/` — stub behind a build tag; every method
  returns `errors.New("doltserver backend not implemented")`. Exists to keep
  the interface honest.

```go
package store

import (
    "context"
    "time"
)

type Status string

const (
    StatusNew       Status = "new"
    StatusClaimed   Status = "claimed"
    StatusDone      Status = "done"
    StatusCancelled Status = "cancelled"
)

type Task struct {
    ID          string
    Title       string
    Description string
    Status      Status
    Priority    int
    CreatedAt   time.Time
    UpdatedAt   time.Time
}

type TaskPatch struct {
    Title       *string
    Description *string
    Status      *Status
    Priority    *int
}

type ListFilter struct {
    Statuses []Status
    Priority *int
    Limit    int
}

type Store interface {
    Open(ctx context.Context, dbPath string) error
    Close() error
    Migrate(ctx context.Context) error
    SchemaVersion(ctx context.Context) (int, error)

    // Task CRUD
    CreateTask(ctx context.Context, t Task) (Task, error)
    GetTask(ctx context.Context, id string) (Task, error)
    UpdateTask(ctx context.Context, id string, p TaskPatch) (Task, error)
    ListTasks(ctx context.Context, f ListFilter) ([]Task, error)

    // Atomic conditional update: new|claimed -> claimed.
    // Returns ErrNotClaimable for done|cancelled.
    Claim(ctx context.Context, id string) (Task, error)

    // Edges. Variadic, transactional, reject self-block and cycles.
    Block(ctx context.Context, id string, blockers ...string) error
    Unblock(ctx context.Context, id string, blockers ...string) error
    UnblockAll(ctx context.Context, id string) (removed int, err error)
    Deps(ctx context.Context, id string) (incoming, outgoing []string, err error)

    // Ready set, sorted prio ASC, created_at ASC.
    Ready(ctx context.Context, limit int) ([]Task, error)

    // Raw passthrough for `autosk sql`.
    QueryRaw(ctx context.Context, q string, args ...any) (Rows, error)
    ExecRaw(ctx context.Context, q string, args ...any) (Result, error)
}
```

### 6.1 Errors

```go
var (
    ErrNotFound        = errors.New("task not found")
    ErrNotClaimable    = errors.New("task is not in a claimable state")
    ErrSelfBlock       = errors.New("a task cannot block itself")
    ErrCycle           = errors.New("edge would create a cycle")
    ErrBlockerNotFound = errors.New("blocker task not found")
    ErrInvalidStatus   = errors.New("invalid status value")
    ErrInvalidPriority = errors.New("priority must be in 0..3")
)
```

### 6.2 SQL dialect strategy

doltlite is SQLite-compatible; dolt-server is MySQL-compatible. The `Store`
interface trades only in Go types — **all SQL stays inside each impl**. No
shared SQL strings across backends. Shared code lives above the interface.

## 9. Concurrency model

- doltlite is single-writer by file lock. Multiple `autosk` invocations
  serialize on this lock.
- `Claim` is implemented as a conditional UPDATE inside a transaction:
  ```sql
  UPDATE tasks
     SET status = 'claimed', updated_at = ?
   WHERE id = ?
     AND status IN ('new', 'claimed')
  ```
  A row-count of 0 means the task is `done` or `cancelled` → return
  `ErrNotClaimable`. A row-count of 1 covers both transitions (the idempotent
  no-op when already `claimed`, and the new→claimed promotion).
- `Block` / `Unblock` run inside a single transaction over all variadic ids;
  partial failure rolls back the whole set.

## 11. Acceptance scenarios

A vetted v0.1 MVP must pass all six end-to-end:

1. **Fresh project.** `cd /tmp/x && autosk create "do the thing"` auto-creates
   `.autosk/db`, applies the initial migration, and prints the new id.
2. **Agent loop.** `autosk ready --json` returns the top task. `autosk claim <id>`
   succeeds; a second `claim` is a no-op (idempotent). `autosk done <id>`
   closes it.
3. **Blocking.** Create A and B; `autosk block B A`; `autosk ready` returns A
   only. `autosk done A` → `autosk ready` now returns B. `autosk show B` shows
   `blocked: false`, `blocked_by: [as-A]` (the edge is historical; the flag
   is computed).
4. **Race-safety.** Two parallel `autosk claim <id>` invocations against a
   task in `new`: exactly one transitions to `claimed`, the other observes
   the idempotent no-op. No SQL error surfaces.
5. **Migrations.** Opening an empty `.autosk/db` and opening a v0.1 db both
   end up at the same `schema_version`. `autosk migrate` is a no-op when
   current. Pending migrations apply in order, each in its own transaction.
6. **JSON contract.** `autosk list --json` and `autosk show <id> --json`
   produce the shape in §10.2. Field order is documented; renames/removals
   are breaking changes.

## 12. Implementation order (suggested)

Roughly vertical, scenario-driven:

1. `go.mod`, cobra skeleton, `cmd/autosk/main.go` with `version`.
2. `internal/store/` interface + errors. Compile-time impl check for both
   backends (`var _ Store = (*doltlite.Store)(nil)`).
3. `internal/projectdb/` walk-up + auto-init.
4. `internal/migrations/` embed + runner; `001_init.sql`.
5. doltlite impl: open/close, migrate, `CreateTask`, `GetTask`.
6. `autosk create`, `autosk show` → scenario 1 green.
7. `ListTasks`, `Ready`, `Claim`. → scenarios 2 & 4 green.
8. `Block`, `Unblock`, `Deps`, derived blocked flag. → scenario 3 green.
9. `autosk update`, `cancel`, `reopen`, `done`.
10. `autosk sql` (read-only by default), `autosk migrate` cli, `autosk history`
    stub.
11. JSON renderer + golden tests → scenario 6 green.
12. doltserver stub file (so the build wires both backends).
13. README + AGENTS.md.
