# `autosk lazy` — TUI dashboard design

**Status:** UI/UX design draft (no implementation tasks yet). This document
locks the layout, navigation, and entity surface. A second plan
(`20260520-Lazy-Impl-Plan.md` or similar) will translate this into a
work breakdown.

`autosk lazy` is a lazygit-style terminal dashboard for the autosk world:
**tasks, jobs, workflows, and agents** in one process, with a fullscreen
job inspector that fuses the existing `autosk attach` live mirror with
an offline `session.jsonl` archive view.

It does **not** replace any subcommand. Everything `lazy` does is also
reachable through the existing CLI; `lazy` is just a denser, faster
front door.

---

## 1. Locked decisions (from the design interview)

| Decision | Locked value |
|---|---|
| Top-level layout | Lazygit-clone: stacked left panels + one wide right detail pane + command log. |
| Left panel set | `[1] Tasks  [2] Jobs  [3] Workflows  [4] Agents`. No dedicated Status panel — project / daemon state lives in the bottom bar. |
| Cross-linking | Lazygit-style scoping: a selection in one panel narrows the others (Tasks → Jobs/Comments; Workflow → Tasks). `*` (or equivalent) widens scope back. |
| Job inspector | **Fullscreen** on `Enter`; `Esc` returns to the dashboard. Four tabs: **Live / Archive / Meta / Signals**. Default = **Live** for non-terminal runs, **Archive** for terminal runs. |
| Project scope | Current cwd only (`.autosk/db` discovered like the rest of the CLI). No `--all-projects` in v1. |
| Writes in v1 | Yes — `claim / done / cancel / enroll / comment add / block / unblock` and the equivalents for jobs (`cancel job`) are hotkey-bound. |
| Search / palette | Yes — `/` filters the focused panel; `:` opens a command palette. |

### Region budget (rough)

| Region | Width | Height |
|---|---|---|
| Left stack | ~30% (min 28 cols, max 40 cols) | full height − bottom bar (1 line) |
| Right detail | remaining width | (full height − 1) × 0.75 |
| Command log | same width as right detail | (full height − 1) × 0.25, min 4 lines, collapsible with `@` |
| Bottom bar | full width | 1 line |

Sub-panel heights inside the left stack split the available vertical
space evenly **but** the focused panel grows by ~40% so its current
selection is always visible. Lazygit does this; we mimic it.

### Bottom bar contents (left → right)

1. `autosk` + project root (truncated to fit).
2. `daemon=ok|down|stale` — health of the per-host UDS daemon.
3. `workers=N`, `q=N`, `r=N` from `/v1/healthz` for this project.
4. **Current scope chip(s).** E.g. `scope: task=as-a1b2` when Jobs is
   filtered by a Tasks selection; `scope: wf=feature-dev` when Tasks
   is filtered by a Workflow selection. `scope: —` when unscoped.
5. `?=help`.

## 3. Left panels — content & per-panel contract

Every left-panel row prints in a fixed columnar format chosen so the
**first column is the id** and the **status glyph is always at column
~10** — this keeps `j/k` scrolling visually consistent across panels.

### 3.1 `[1] Tasks`

| Column | Source |
|---|---|
| `id` | `tasks.id` |
| status glyph | one of `● ⏳ ✋ ✓ ✗` for `new / in_workflow / human_feedback / done / cancelled` |
| priority | `P{n}` |
| step ptr | `▶stepName` when `status='in_workflow'`, blank otherwise |
| title | truncated |

Default filter: `status IN ('new','in_workflow','human_feedback')`
(matches `autosk list` default). `/` opens a fuzzy filter; `\` cycles
status filters (`open | all | done | cancelled | …`).

Cross-link consumer: the **right detail** shows the selected task's
description, current step, recent jobs, comments, blockers, and the
last `n` step_signals.

Cross-link producer: a Tasks selection adds `scope: task=<id>` and
narrows panel `[2] Jobs` to `task_id = <id>`. `*` widens.

### 3.2 `[2] Jobs`

| Column | Source |
|---|---|
| `job_id` | `daemon_runs.id` |
| status glyph + label | `◐ stream / ◯ done / ✗ failed / · queued / ‣ cancelled` |
| workflow:step | from joined `workflow` / `step` of the run |
| age | `now() - created_at` (human) |
| attach hint | `(N)` clients attached when N>0; `*live*` highlight when the daemon thinks this run is currently streaming |

When `scope: task=<id>` is active the Jobs panel is filtered (header
shows `(filtered by task as-a1b2)`). When `scope: wf=<name>` is
active Jobs are filtered to runs whose step belongs to that workflow.

Cross-link consumer: the **right detail** shows job summary (status,
durations, corrections, error, exit code, session path, PID,
attach_count, streaming) + a transcript preview (last ~20 events from
`/v1/jobs/{id}/messages?limit=20`).

`Enter` on a job opens the **fullscreen Job Inspector** (§5).

### 3.3 `[3] Workflows`

| Column | Source |
|---|---|
| name | `workflows.name` |
| `n steps` | `count(steps)` |
| step-load | `first_step(n_tasks)/next_step(n_tasks)` micro-histogram |

Synthetic `single:<agent>` workflows are hidden behind `\` (toggle
`--all`). `Enter`/right-detail shows the step graph + per-step agent
and transition list.

Cross-link producer: selecting a workflow adds
`scope: wf=<name>` and filters Tasks (`workflow_id`) and Jobs.

### 3.4 `[4] Agents`

| Column | Source |
|---|---|
| presence glyph | `●` installed, `!` db_only, `○` builtin |
| name | full npm package name (or `human`) |
| version | from installed package.json |
| source | `builtin|installed|db_only` |

Right-detail shows: model / thinking / first_message preview / extra_args /
pi_extensions / pi_skills / custom runner path / list of tasks
currently owned (by author_id or current_step.agent_id).

Cross-link: selecting an agent does **not** auto-filter Tasks (would
be confusing — `author` and `current_step.agent` are different
concepts). Instead, `Enter`-on-agent flips Tasks to a manual filter
chip the user can confirm: `scope: agent=<name> (press * to clear)`.

## 4. Right detail (dashboard mode)

The right pane re-renders whenever the focused selection changes. It
is **read-only** in this mode — all writes go through hotkeys, not
through clicking inside the detail.

For each entity kind:

| Focused panel | Detail content |
|---|---|
| Tasks | header (id/status/priority/blocked); workflow+step+agent; created/updated; description (full); recent jobs (≤5); comments (≤5); blockers (open) / blocks (open). |
| Jobs | header (job_id/status/streaming/attached/corrections); workflow+step+agent; timings (created/started/finished/duration); error+exit_code; session_path; transcript preview (last 20 events, rendered like the attach TUI). |
| Workflows | header (name/synthetic?); description; first_step; step graph table (`step → next_steps`); per-step agent + task count. |
| Agents | header (name/source/version); model/thinking/first_message_file (path) and first 10 lines preview; extra_args/pi_extensions/pi_skills; runner (if custom); count of tasks owned. |

The detail is intentionally text-only and re-renders are cheap; the
TUI runs a 2s poll loop (same cadence as the daemon poller) to refresh
list data, plus an SSE channel for the focused job's transcript
preview when the daemon is reachable.

## 5. Fullscreen Job Inspector

`Enter` on a Jobs row hides the dashboard and opens a near-fullscreen
inspector. `Esc` (or `Ctrl-O`) restores the dashboard with the same
job still selected.

```
┌ run-9ab1 · feature-dev ▶ dev · running · streaming · attached:1 · corr:0/3 ────┐
│ [Live] [Archive] [Meta] [Signals]                                              │
├────────────────────────────────────────────────────────────────────────────────┤
│ user:     please refactor the token validation into its own pkg                │
│                                                                                │
│ assistant_thinking:                                                            │
│   Need to look at internal/auth/...                                            │
│                                                                                │
│ tool_call: bash { command: "rg -n TokenValidate internal/auth" }               │
│ tool_result: ...                                                               │
│                                                                                │
│ assistant_text:                                                                │
│   Found three call sites. Extracting into internal/auth/tokenvalidate ...     │
│                                                                                │
│ ──────────────────────────────────────────────────────────────────────────── │
│ ▎ type a message · ctrl+d send · ctrl+f follow_up · ctrl+a abort · ctrl+c quit│
│                                                                                │
├────────────────────────────────────────────────────────────────────────────────┤
│ flash: dispatched: steer (just now)         tip: [/] search, [m] toggle meta   │
└────────────────────────────────────────────────────────────────────────────────┘
```

### 5.1 Tabs

- **`[` / `]`** cycles tabs. `1..4` jumps directly (1=Live, 2=Archive,
  3=Meta, 4=Signals).
- Default tab is computed from the run's status at open time:
  - `running` / `queued` → **Live**.
  - `done` / `failed` / `cancelled` → **Archive**.
  - If the run was **terminalAtConnect** (Live opened on a terminal
    run, e.g. user pressed `1` explicitly), the Live tab degrades to
    a read-only replay — exactly the v2 attach semantics.

### 5.2 Live tab

This is exactly the existing `internal/attach/tui` integrated as a
sub-component. Contract is unchanged:

| Region | Behaviour |
|---|---|
| transcript viewport | SSE `?attach=true&full=true` replay + tail. `Last-Event-ID` resume. |
| textarea | multi-line; `Enter` is newline; `Ctrl-D` send (daemon picks prompt/steer), `Ctrl-F` follow_up, `Ctrl-A` abort, `Ctrl-C/Ctrl-Q` close inspector (= Esc back to dashboard). |
| status line | run status · `streaming|idle` · `attached:N` · `corrections:K/M` · ephemeral flash. |

Attaching takes the daemon's attach counter from 0→1; closing the
inspector drops it back. This is symmetric with `autosk attach` —
exiting the inspector detaches; it does not pause the run.

### 5.3 Archive tab

Read-only renderer of the job's session as it lives on disk:

- Source: `GET /v1/jobs/{id}/messages?full=true` (paginated server-side;
  the TUI streams pages until end).
- No SSE, no textarea, no input dispatch — purely scrollback.
- `/` opens an incremental search across the rendered transcript;
  `n`/`N` jump between matches.
- `y` copies the selected event's `data:` blob to the OS clipboard
  (best-effort; falls back to a flash with the file path).
- `g` / `G` jump to top / bottom; `Ctrl-F`/`Ctrl-B` page; `j`/`k`
  line scroll.
- When a job is still running and the user opens Archive explicitly,
  the tab shows a footer hint `Archive view — does not auto-refresh
  (press R to reload, or [ for Live)`.

### 5.4 Meta tab

A static key/value sheet rendered from `api.JobResponse`:

```
job_id           run-9ab1
task             as-a1b2  (refactor the auth flow)
workflow:step    feature-dev:dev
agent            @autosk/developer
status           running
streaming        true
attached         1 (you)
corrections      0 / 3
pi_session_id    01jzv5...
session_path     .autosk/sessions/run-9ab1/session.jsonl
pid              71234
created          2026-05-19T16:22:08Z   (5m ago)
started          2026-05-19T16:22:09Z   (5m ago)
finished         —
duration         00:05:14
error            —
exit_code        —
```

Read-only; press `Enter` on `task` to jump back to the dashboard with
that task selected; press `Enter` on `workflow:step` to jump to the
workflow.

### 5.5 Signals tab

Two stacked sub-regions:

1. **Step signals** for this run (the rows the agent inserted via
   `autosk step next`). PK enforces "exactly one", but during
   correction loops earlier failed runs may have related rows; show
   all rows whose `run_id` matches.
2. **Comments** on the underlying task, scoped to those added
   `since started_at` (delimits "comments observed during this run").

Useful when an agent failed `agent_did_not_emit_transition` — you
see at a glance whether a signal was inserted late or never.

## 6. Navigation reference

### 6.1 Global

| Key | Action |
|---|---|
| `1` … `4` | Focus left panel by number. |
| `Tab` / `Shift-Tab` | Cycle left panels. |
| `0` | Focus the right detail (enables `j/k` scrolling in detail). |
| `Enter` | Drill into selection. On Jobs → fullscreen inspector. On other panels → right detail is already showing details, so `Enter` cross-links: Tasks→focus Jobs filtered to that task; Workflows→focus Tasks filtered to that wf; Agents→prompt-confirm Tasks filter. |
| `Esc` | Pop one level: inspector → dashboard; filter chip → unscoped; palette → close. |
| `?` | Help overlay (key-binding cheatsheet, scrollable). |
| `:` | Command palette (§6.3). |
| `/` | Filter the focused panel (incremental, case-insensitive). |
| `*` | Clear scope chips (widen everything). |
| `R` | Refresh the focused panel (force re-fetch, ignore poll cadence). |
| `@` | Toggle command-log visibility. |
| `q` / `Ctrl-C` | Quit. |

### 6.2 Per-panel hotkeys

**Tasks `[1]`**

| Key | Action |
|---|---|
| `n` | New task — opens prompt for title (multi-line via `$EDITOR` on `e`). |
| `c` | Claim selected task (`autosk assign --agent human` when on `new`, or `claim`-equivalent in current API). |
| `d` | Mark **done**. Confirms when status was `in_workflow`. |
| `x` | Cancel. Confirms. |
| `e` | Enroll into workflow/agent (sub-prompt picks the workflow or agent). |
| `r` | Resume (`human_feedback` → `in_workflow`); sub-prompt optionally picks `--to step`. |
| `b` | Add a blocker (id picker from Tasks list). |
| `u` | Remove a blocker (list of open blockers; `--all` via capital `U`). |
| `m` | Add a comment (opens `$EDITOR`, then `autosk comment add`). |
| `p` | Set priority (`0..3` picker). |
| `o` | Reopen (`done`/`cancelled` → `new`). |
| `J` / `K` | Scroll detail viewport. |

**Jobs `[2]`**

| Key | Action |
|---|---|
| `a` | Open inspector at **Live** tab (same as `Enter`). |
| `s` | Open inspector at **Archive** tab. |
| `i` | Open inspector at **Meta** tab. |
| `K` | Cancel (`DELETE /v1/jobs/{id}`); confirms. |
| `m` | Tail last 20 messages into command log (no inspector). |

**Workflows `[3]`**

| Key | Action |
|---|---|
| `n` | Create from file — prompts for a path (then `autosk workflow create --file`). |
| `v` | View raw JSON in the detail pane (toggle). |
| `D` | Delete (confirms). |
| `\` | Toggle synthetic visibility. |

**Agents `[4]`**

| Key | Action |
|---|---|
| `i` | Install (prompts for npm name + optional `--version`). |
| `u` | Uninstall (confirms, `--force` via `U`). |
| `R` | `autosk agent runtime install`. |

### 6.3 Command palette (`:`)

A flat fuzzy-searchable command list backed by the per-panel hotkey
table. Examples that surface regardless of focus:

```
:task new
:task enroll <id> --workflow feature-dev
:job cancel <job-id>
:workflow create --file path/to/wf.json
:agent install @autosk/developer
:filter status=done
:goto as-a1b2     (= jump-to-id, also bindable to `g`)
:scope clear
:quit
```

Palette executes the verb via the same code paths the hotkeys use;
unsuccessful commands print to the command log with a red flash.

### 6.4 Filter language (`/`)

Per-panel quick-filter. Typing `as-a1` filters by id substring.
Prefix-driven facets are supported per panel:

- Tasks: `p:1 status:in_workflow wf:feature-dev step:dev agent:dev`
- Jobs: `status:running wf:feature-dev task:as-a1b2`
- Workflows: `synthetic:true`
- Agents: `source:installed`

Filter chips render in the bottom bar next to scope chips; `Esc`
clears the panel's filter without quitting.

## 7. Cross-linking — exact rules

The "lazygit-style scoping" decision implies a precise contract. We
spell it out so future edits don't drift.

| Trigger | Effect |
|---|---|
| Move cursor in Tasks `[1]` | Right detail re-renders for that task. Jobs panel applies **scope chip** `task=<id>` and filters to `daemon_runs.task_id = <id>`. Comments + signals in the right detail are scoped to that task too. |
| Move cursor in Workflows `[3]` | Right detail re-renders for that workflow. Tasks panel applies scope chip `wf=<name>` and filters to `tasks.workflow_id = (wf.id)`. Jobs panel applies the same. |
| Move cursor in Agents `[4]` | Right detail re-renders for that agent. **No automatic scope chip.** Press `Enter` to opt into `agent=<name>` filtering Tasks (because the relation is ambiguous — author vs. current_step agent). |
| Move cursor in Jobs `[2]` | Right detail re-renders for that job. No scope chip propagates back. |
| `*` (anywhere) | Clears all scope chips. |
| `Esc` inside an active filter chip | Drops that one chip. |

Scope is **additive**: a `wf=X` chip plus a `task=Y` chip means
`task.workflow=X AND job.task=Y`. Conflicts (e.g. task whose wf
differs from the chip) just produce an empty filtered list with a
muted "0 / N rows after scope".

## 8. Command log

Bottom-right block, lazygit-style. Three line classes:

| Class | Examples | Style |
|---|---|---|
| info | `enroll as-a1b2 ok`, `dispatched: steer`, `cancelled job run-9ab1` | default |
| warn | `agent_did_not_emit_transition`, `daemon unreachable` | yellow |
| err  | `409 runner not registered`, `500 daemon error` | red |

Lines are prepended with a relative timestamp (`16:42` or `2s ago`).
`@` toggles visibility; when hidden, a one-line breadcrumb at the
bottom of the right detail shows the last log line.

Persisted in-memory only (no on-disk command log file). The bottom
"random tip" line cycles the same hint pool as lazygit-style help
text (`'a' attach`, `'/' filter`, `':' commands`, …).

## 9. Daemon dependency & graceful degradation

The TUI works in three observable states:

| State | Source | Effect |
|---|---|---|
| **daemon ok** | `GET /v1/healthz` returns 200 | Jobs panel populates from `GET /v1/jobs`; Live tab in inspector opens SSE; status bar shows `daemon=ok workers=N q=N r=N`. |
| **daemon down** | UDS dial fails | Jobs panel falls back to reading `daemon_runs` directly from `.autosk/db`. Live tab is disabled with a flash `daemon unreachable (run \`autosk daemon serve\` to enable)`. Archive tab still works (reads `session_path` from the DB). Status bar shows `daemon=down`. |
| **daemon stale** | UDS dials but `/v1/healthz` 5xx | Same as down but status bar shows `daemon=stale` so the user knows the socket file is suspicious. |

This contract means `autosk lazy` is **useful without the daemon**: you
can read tasks, browse archived jobs, and edit task state. Only Live
attach and `cancel job` require a live daemon.

## 10. What v1 does **not** include

- `--all-projects` cross-project view. Scope is cwd.
- Mouse support (Bubble Tea-style — keyboard-only v1).
- Resizable panels via drag. Width-split is fixed (~30% left).
- Multi-select / bulk actions.
- A history pane reading `audit_log`. (Audit log itself is not
  implemented — see README roadmap.)
- A "compose new workflow" form. `:workflow create --file path` only.
- Customisable keymaps. We pick one set and document it.
- Reconnect/backoff in the SSE client. Same constraint as `attach v2`.

These are explicitly tracked so the implementation plan can put
them under §"open items".

## 11. Open questions for implementation phase

These are **not** UX decisions — they're routing/refactor choices that
should be resolved when writing the impl plan, not now:

1. Where does the lazy TUI live in the tree? Suggested:
   `internal/lazy/tui` (new package) reusing
   `internal/attach/tui` as the embedded Live-tab component.
2. Does the inspector reuse `internal/daemon/client` directly, or do
   we introduce a thin `internal/lazy/datasource` layer that hides
   "daemon vs. direct DB" behind a single interface? (Probably yes —
   see §9 — but the boundary is a design call.)
3. How are writes wired? Two options:
   - call into existing `internal/store` and friends directly (one
     binary, one transaction surface);
   - shell out to the user's `autosk` (idempotent, slow, robust).
   For v1 the bias is direct in-process calls because we already
   have the store dependency wired by virtue of being inside the
   same binary, but the abstraction in §11.2 makes the choice
   reversible.
4. Polling cadence and resource budget — likely 2s for list panels
   plus SSE-on-focus for the selected job's transcript preview. A
   `--refresh-interval` flag is a v1 convenience.
5. Test strategy — bubbletea models are unit-testable; we should
   reuse the patterns already in `internal/attach/tui/*_test.go`.

These all become tasks in the implementation plan; calling them out
here so the design doc isn't blamed for the silence.

---

## 12. Out of scope (explicitly)

- Replacing `autosk attach`. It stays as the headless / scripted
  entry point and as the Live-tab component of `lazy`. The two
  commands coexist forever.
- Replacing or competing with the daemon's HTTP API surface. `lazy`
  is a consumer.
- Any web-UI variant. Terminal-only.
