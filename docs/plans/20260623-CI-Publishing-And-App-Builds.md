**Status:** plan (analysis + a local signing spike; no CI/code changes yet).

**Owners:** autosk CI / release.

**Related files:**

## 1. Decisions (locked)

| # | Decision | Choice |
|---|----------|--------|
| D1 | npm + Docker publish trigger | **After CI succeeds on `main`**, version-gated (a package publishes only when its `package.json` version is bumped; images push every run) — see R7 |
| D2 | macOS GUI delivery | **One Homebrew Cask** named `autosk`: a signed + notarized `autosk.app` that **embeds** the CLI (`autosk`) + TUI + daemon (`autoskd`); the cask symlinks them onto `PATH` |
| D3 | Apple signing scope | **Full** — paid Apple Developer account: Developer ID signing + notarization (macOS), App Store Connect signing + TestFlight (iOS) |
| D4 | macOS / iOS app-build trigger | **On semver tag** (extend `release.yml`) |
| D6 | macOS architecture | **Apple Silicon only** (`aarch64-apple-darwin`) |
| R1 | macOS signing feasibility | **Verified locally** (see §10): the real `autoskd` Bun binary signs + runs under hardened runtime; entitlements wired |
| R5 | Homebrew formula | **Dropped.** Tap is **cask-only** (macOS). Linux is no longer served via Homebrew — it ships as **GitHub Release assets** (binaries + GUI AppImage/.deb). *Supersedes the earlier "keep Linux formula + GUI-via-formula" (D5/D7).* |
| R6 | Pre-release policy | **TestFlight** uploads on **every** tag incl. `-rc`/`-beta`; the **Cask** bumps on **stable tags only** |
| R7 | Publish gating | **`publish.yml` runs via `workflow_run` after the `CI` workflow succeeds on `main`** (a red build never publishes) |
| S1 | npm auth | **OIDC Trusted Publishers** — no `NPM_TOKEN` at all (see §3, §9) |
| S2 | Apple secrets | **One shared App Store Connect API key** for iOS signing + macOS notarization + TestFlight upload (see §9) |

Resulting triggers:

```
push → main          : CI checks (ci.yml, gui.yml)  →  on CI success: publish.yml (npm via OIDC + GHCR images)
push → tag vX.Y.Z    : release.yml — binaries + GitHub Release + macOS cask(stable only) + Linux assets + iOS→TestFlight(all tags)
```

## 2. Current state (baseline)

- **`ci.yml`** — daemon (Bun typecheck+test), Go (build+test, ubuntu + macos-26), clean-layout smoke. Push-`main` + PR. Checks only.
- **`gui.yml`** — GUI front end (`npm ci`, typecheck+IPC guard, vite build, vitest) + `cargo check`. Push-`main` + PR. No bundling.
- **`release.yml`** — on `v*` tag: `autosk` (Go) + `autoskd` (Bun compile) for `linux-x64` + `osx-arm64`, GitHub Release (notes from `CHANGELOG.md`), then **bumps `Formula/autosk.rb`** (CLI+daemon; pre-releases skip).
- **npm + Docker** — only via `scripts/publish-extensions.sh`, by hand. It already: runs daemon + pi-tools sanity, **skips any `@autosk/*` whose exact version is already on npm**, publishes 9 packages in dependency order, and buildx-pushes `pi-runtime` + `claude-runtime` to GHCR multi-arch. Idempotent by construction.
- **All 9 packages already carry a `repository` field with `directory`** (verified) → trusted-publishing provenance works with no package.json changes.
- **GUI signing** — none today. `gen/apple/` is gitignored (regenerated by `tauri ios init`); its `ExportOptions.plist` is `debugging`.

The script's idempotency is the keystone: D1 reduces to "run the existing script after CI is green."

## 3. Workstream A — publish npm (OIDC) + Docker on `main`, gated on CI

### A.1 `.github/workflows/publish.yml` (triggered after CI)

```yaml
name: Publish (npm + images)
on:
  workflow_run:
    workflows: ["CI"]          # ci.yml's `name:`
    types: [completed]
concurrency: { group: publish-main, cancel-in-progress: false }
permissions:
  contents: read
  packages: write              # GHCR push via GITHUB_TOKEN
  id-token: write              # OIDC for npm trusted publishing
jobs:
  publish:
    # R7: only a green CI on a push to main publishes.
    if: >-
      github.event.workflow_run.conclusion == 'success' &&
      github.event.workflow_run.head_branch == 'main' &&
      github.event.workflow_run.event == 'push'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
        with: { ref: ${{ github.event.workflow_run.head_sha }} }   # publish the exact commit CI validated
      - uses: oven-sh/setup-bun@v2
        with: { bun-version: latest }
      - uses: actions/setup-node@v6
        with: { node-version: "24" }            # NO registry-url / token needed under OIDC
      - run: npm install -g npm@latest          # trusted publishing needs npm >= 11.5.1
      - uses: docker/setup-qemu-action@v3
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with: { registry: ghcr.io, username: ${{ github.actor }}, password: ${{ secrets.GITHUB_TOKEN }} }
      - name: publish (version-gated npm via OIDC + GHCR images)
        env: { IMAGE_TAG: latest }
        run: scripts/publish-extensions.sh --publish
```

### A.2 npm OIDC Trusted Publishers (S1 — no token)

Research-confirmed requirements (npm docs, GA 2025-07-31):

- **No `NPM_TOKEN`.** The workflow's `id-token: write` lets npm mint a short-lived OIDC credential; npm matches it against a per-package **trusted publisher** config.
- **npm CLI ≥ 11.5.1, Node ≥ 22.14.0** (we `npm i -g npm@latest`; local check shows 11.9.0 already works).
- **One-time on npmjs.com, per package:** add a trusted publisher → org/user `wierdbytes`, repo `autosk`, workflow file `publish.yml` (optional environment). Must be done for **all 9** `@autosk/*` packages (the script publishes them all).
- **Provenance is automatic** under trusted publishing (no `--provenance` flag); it validates against each package's `repository` field — already present.
- **Cloud-hosted runners only** (we use GitHub-hosted ✓).
- The publish **script is unchanged**: `npm publish --access public` auto-detects OIDC in Actions; `registry_has_version()` keeps the version-gate. A push that bumps no version is a no-op per package.

> `workflow_run`-triggered jobs still produce an OIDC token whose `job_workflow_ref` points at `…/publish.yml`, so the trusted-publisher match holds. (Validate on first run; this is the one OIDC interaction worth watching.)

### A.3 Images & remaining knobs

- GHCR via built-in `GITHUB_TOKEN` + `packages: write`. First push per image may need the package linked to the repo (or a `CR_PAT` if org policy blocks `GITHUB_TOKEN`).
- Images aren't version-gated → rebuilt+pushed each run (buildx cache keeps it cheap). Default tag `latest`; recommend **also** `:sha-<short>` for rollbacks (one-line loop in the script; honours `IMAGE_TAG`). Optionally `paths`-filter the image step to `daemon/extensions/*/docker/**`.
- Keep the script's own `bun` sanity (it also covers **pi-tools**, which `ci.yml` does not).

## 4. Workstream B — macOS Cask: bundled, signed, notarized (on stable tag)

Goal: `brew install --cask wierdbytes/autosk/autosk` installs a notarized `autosk.app` whose bundle contains `autosk` (CLI/TUI) and `autoskd`; the cask symlinks both onto `PATH`.

### B.1 Why embed the binaries

A Finder/Dock-launched `.app` gets a **minimal `PATH`** (no `/opt/homebrew/bin`). The GUI auto-spawns `autoskd` in local mode, so `autoskd` must be resolvable without a login shell → embed it in the `.app` and resolve relative to the executable (a Homebrew-only symlink works for a terminal CLI, not a Finder GUI). Hence D2 "GUI bundles CLI+TUI+daemon."

### B.2 Tauri `externalBin` (sidecars) + entitlements

`gui/src-tauri/tauri.conf.json`:

```jsonc
"bundle": {
  "externalBin": ["binaries/autosk", "binaries/autoskd"],
  "macOS": { "entitlements": "entitlements.plist", "hardenedRuntime": true }
}
```

CI stages the arm64-only sidecars Tauri expects:

- `gui/src-tauri/binaries/autosk-aarch64-apple-darwin`   ← `make build`
- `gui/src-tauri/binaries/autoskd-aarch64-apple-darwin`  ← `scripts/package-autoskd.sh`

`entitlements.plist` (validated in §10) carries the Bun JIT entitlements:

```xml
<key>com.apple.security.cs.allow-jit</key><true/>
<key>com.apple.security.cs.allow-unsigned-executable-memory</key><true/>
<key>com.apple.security.cs.disable-library-validation</key><true/>
<key>com.apple.security.cs.allow-dyld-environment-variables</key><true/>
```

**Action item:** make the GUI backend prefer the embedded `autoskd` sidecar over bare-`PATH` resolution when present (small change in `gui/src-tauri`; or use `sidecar("autoskd")`).

### B.3 Signing / notarization (Tauri env)

- `APPLE_CERTIFICATE` (base64 Developer ID Application `.p12`) + `APPLE_CERTIFICATE_PASSWORD`
- `APPLE_SIGNING_IDENTITY` (`Developer ID Application: … (TEAMID)`)
- **Notarization via the shared App Store Connect API key (S2):** `APPLE_API_KEY` (key id), `APPLE_API_ISSUER`, `APPLE_API_KEY_PATH` (the `.p8`, materialised from a base64 secret into `./private_keys/AuthKey_<id>.p8`)
- `APPLE_TEAM_ID`

CI keychain: create an ephemeral keychain, `security import` the `.p12` (unlock via `KEYCHAIN_PASSWORD`), then `tauri build --target aarch64-apple-darwin`.

### B.4 Risk that remains after the §10 spike

The §10 spike proved the **mechanics** (real `autoskd` runs under hardened runtime; entitlements embed cleanly; ad-hoc `flags=…runtime`). Two things the spike could **not** cover, so keep the mitigation:

1. **Tauri bundler signing of sidecars** ([tauri#11992](https://github.com/tauri-apps/tauri/issues/11992)) — the bundler has historically failed to hardened-runtime-sign `externalBin` entries, breaking notarization. **Mitigation:** before notarizing, explicitly `codesign --force --options runtime --entitlements entitlements.plist --sign "$APPLE_SIGNING_IDENTITY"` the two embedded binaries inside `autosk.app/Contents/MacOS/`, then notarize (`xcrun notarytool submit … --key … --key-id … --issuer …`) + `xcrun stapler staple`.
2. **Apple notary acceptance** — only verifiable with the real Developer ID cert (ad-hoc can't be notarized). First green run is the acceptance test; do the M2 spike on a branch with the real cert before wiring the whole job.

### B.5 Outputs + cask render

1. Signed, notarized, stapled `autosk_<ver>_aarch64.dmg` → **GitHub Release asset**.
2. **Render + push `Casks/autosk.rb`** to the tap (reuse `HOMEBREW_TAP_TOKEN`), **stable tags only** (R6):
   ```ruby
   cask "autosk" do
     version "X.Y.Z"
     sha256 "<dmg-sha>"
     url "https://github.com/wierdbytes/autosk/releases/download/v#{version}/autosk_#{version}_aarch64.dmg"
     name "autosk"
     depends_on macos: ">= :big_sur"
     app "autosk.app"
     binary "#{appdir}/autosk.app/Contents/MacOS/autosk"
     binary "#{appdir}/autosk.app/Contents/MacOS/autoskd"
     zap trash: ["~/.autosk", "~/Library/Application Support/com.autosk.gui"]
   end
   ```
   (Confirm whether Tauri keeps/strips the `-aarch64-apple-darwin` suffix in `Contents/MacOS/` and fix the `binary` paths accordingly.)

## 5. Workstream C — Linux GUI + binaries as Release assets (no Homebrew)

Per **R5** the formula is dropped, so Linux is served by **GitHub Release assets only**:

- Existing `autosk` + `autoskd` `linux-x64` binaries (already built by `release.yml`).
- **GUI:** `tauri build` on an `ubuntu` runner (same deps as `gui.yml`: `libwebkit2gtk-4.1-dev`, `libgtk-3-dev`, `libayatana-appindicator3-dev`, `librsvg2-dev`, `libsoup-3.0-dev`) → upload `autosk_<ver>_amd64.AppImage` (and optionally the `.deb`) as Release assets.
- No signing needed on Linux.

> Linux GUI is `linux-x64` only (matches D6's macOS-arm64-only stance; Linux-arm64 is out of scope until a runner exists). Install docs point Linux users at the Release page / npm rather than `brew`.

## 6. Workstream D — iOS GUI → TestFlight (every tag)

Mirrors `scripts/install-gui-ios.sh`, but App-Store-signed and auto-uploaded, on a `macos` runner.

### D.1 Build

1. `npm ci` in `gui/`; add the `aarch64-apple-ios` Rust target.
2. `tauri ios init` — `gen/apple/` is **gitignored**, so CI regenerates it from `tauri.conf.json` each run. **Consequence:** every iOS customization (icons, `Info.plist`, capabilities, bundle id `com.autosk.gui`) must live in config, never hand-edited in the generated project. Audit current local `gen/apple` customizations before first CI run.
3. `tauri ios build --export-method app-store-connect` → `gen/apple/build/arm64/autosk.ipa`.

### D.2 Automatic signing (S2 — no manual cert/profile)

Research-confirmed: Tauri iOS **automatic** signing in CI needs only the **App Store Connect API key** — `APPLE_API_ISSUER`, `APPLE_API_KEY`, `APPLE_API_KEY_PATH` (+ `APPLE_DEVELOPMENT_TEAM`). This **eliminates** the manual `IOS_CERTIFICATE` + `IOS_MOBILE_PROVISION` path and **reuses the same key** as macOS notarization + the TestFlight upload. One-time portal setup: register App ID `com.autosk.gui` and create the app record in App Store Connect.

### D.3 Monotonic build number

TestFlight rejects a re-used `CFBundleVersion`. Inject a monotonic value (`github.run_number`) as the iOS build number each run. **Action item:** confirm the Tauri knob for iOS `CFBundleVersion` and wire `github.run_number`.

### D.4 Upload (all tags, incl. pre-release)

`xcrun altool --upload-app --type ios --file …/autosk.ipa --apiKey "$APPLE_API_KEY" --apiIssuer "$APPLE_API_ISSUER"` (same shared key). Per R6 this runs for **every** tag — TestFlight is the beta channel — while the Cask bump (B.5) is stable-only.

The tap `wierdbytes/homebrew-autosk` becomes **cask-only**: add `Casks/autosk.rb`, **retire `Formula/autosk.rb`**. With no same-named formula present, `brew install autosk` (after `brew tap wierdbytes/autosk`) resolves unambiguously to the cask — no `--formula`/`--cask` collision. ## 10. R1 — macOS signing spike (done locally, 2026-06-23)

Ran on this Apple-Silicon host (macOS 25.5, `bun` 1.3.14, `codesign`, npm 11.9.0):

1. **Compiled the real artifact** — `bun build --compile core/src/index.ts` → a 63 MB `autoskd` single-file binary (the exact release output).
2. **Hardened-runtime signing works** — `codesign --force --options runtime --entitlements entitlements.plist --sign -` succeeds; `codesign --verify` → *valid on disk / satisfies its Designated Requirement*; `codesign -dv` shows `flags=0x10002(adhoc,runtime)`; all four JIT entitlements embed.
3. **The signed hardened-runtime binary still executes** (reaches its socket/"already running" logic).
4. **Plain Bun JIT is *not* blocked** even without the entitlements — a JIT-stress binary exits `0` under hardened runtime with and without entitlements. The entitlements are a harmless, recommended belt-and-suspenders rather than a hard blocker for this Bun version.
5. **A scary "hang" was a test artifact, not signing** — `autoskd <anything>` with a free socket simply *becomes* the long-running daemon (there is no early-exit `version`/`--help` subcommand). Unrelated to entitlements.

**Conclusion:** the embed-and-sign approach is mechanically sound. The residual unknowns are **bundler sidecar signing** (#11992) and **Apple notary acceptance**, both addressed by B.4's explicit `codesign` pass + an M2 spike with the real Developer ID cert. (Minor follow-up: add an early-exit `autoskd version`/`--help` so smoke tests don't accidentally launch the daemon.)

## 11. Sequencing / milestones

1. **M1 — `publish.yml` (Workstream A).** Configure the 9 trusted publishers on npmjs.com, ship the `workflow_run`-gated workflow. Verify a deliberate version bump publishes (with provenance) and an unrelated push is a no-op. Lowest risk, immediate value.
2. **M2 — macOS signing spike with the real cert (B.4).** Build + notarize one DMG end-to-end (incl. the explicit sidecar `codesign` pass) before wiring the release job. Riskiest unknown.
3. **M3 — macOS cask job (Workstream B).** Wire build→sign→notarize→asset→cask-bump into `release.yml`; make the GUI prefer the embedded `autoskd`.
4. **M4 — Linux GUI assets (Workstream C).** AppImage(+.deb) build + Release upload; retire the formula in the tap (§7 migration).
5. **M5 — iOS TestFlight (Workstream D).** Portal/ASC setup, `gen/apple` reproducibility audit, build-number wiring, automatic signing + upload.
6. **M6 — docs/changelog.** README install matrix (macOS cask, Linux assets, npm), `gui/README.md` distribution section, `CHANGELOG.md` entries (§13).

M1 ships independently now; M2 gates M3; M4/M5 parallelize once M3's `release.yml` scaffolding exists.
