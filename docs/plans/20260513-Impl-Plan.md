# autosk v0.1 — Implementation Plan

**Date:** 2026-05-13
**Companion to:** [`20260513-Init-Plan.md`](./20260513-Init-Plan.md) (spec)
**Status:** Ready to execute

This plan turns the locked v0.1 spec into a sequence of small, testable
deliverables. Phases are ordered so each one turns at least one acceptance
scenario from §11 of the spec green.

---

## 0. Ground rules

- **Go 1.24+**, CGO required (link against `libdoltlite`).
- **Driver**: `github.com/mattn/go-sqlite3` with build tag `libsqlite3`. Doltlite
  is a drop-in for SQLite at the C ABI; we point the linker at `libdoltlite.a`
  instead of system `libsqlite3` and reuse the standard driver. Confirmed
  working pattern in `~/me/dev/doltlite/examples/go/main.go`.
- **`db.SetMaxOpenConns(1)`** is mandatory for the doltlite store — Dolt branch
  state and the single-writer lock both live in the connection.
- **Cobra** for CLI (matches beads).
- **No `internal/types` ↔ `internal/store` cycle** — the canonical types live
  in `internal/store` (or a thin `internal/task` that `store` imports).
- **Every phase ends with `make test` green** before moving on.

### 0.1 Doltlite prerequisite

The build assumes `libdoltlite.{a,so,dylib}` plus `sqlite3.h` are available.
For local development we point at `~/me/dev/doltlite/build`. A `Makefile`
exports CGO env vars; users override via `DOLTLITE_DIR`.

```make
DOLTLITE_DIR ?= $(HOME)/me/dev/doltlite/build

export CGO_CFLAGS  := -I$(DOLTLITE_DIR)
export CGO_LDFLAGS := $(DOLTLITE_DIR)/libdoltlite.a -lz -lpthread
GOFLAGS_BUILD       := -tags libsqlite3
```

A `make doctor` target verifies the headers and library exist before invoking
`go build`, with a clear "build doltlite first" error.

## 2. Phase 0 — Bootstrap  *(size: M)*

**Goal:** repo compiles, `autosk version` runs, doltlite link verified.

### Tasks
1. `go mod init autosk` (module path TBD — leave as `autosk` for local use; switch to a public path before release).
2. `.gitignore`, `.gitattributes`, `LICENSE` (MIT or match beads).
3. **`Makefile`** with targets:
   - `build` — `go build -tags libsqlite3 -o bin/autosk ./cmd/autosk`
   - `test` — `go test -tags libsqlite3 ./...`
   - `lint` — `golangci-lint run`
   - `doctor` — check `$(DOLTLITE_DIR)/sqlite3.h` and `libdoltlite.a` exist
   - `clean`
4. `cmd/autosk/main.go` with cobra root command + `version` subcommand.
5. `cmd/autosk/version.go` — prints binary version, schema version (= 0 here), backend.
6. **Smoke test**: `internal/store/doltlite/smoke_test.go` opens an in-memory doltlite db, runs `SELECT doltlite_engine();`, asserts result is `prolly`. This is the canonical "link works" test.
7. `.github/workflows/ci.yml` skeleton (Go build + test; doltlite library is downloaded from doltlite releases, or test is skipped if absent — keep simple for MVP).

### Definition of done
- `make doctor build test` is green on macOS with a locally built doltlite.
- `bin/autosk version` prints something sensible.
- Smoke test confirms the prolly engine is active.

## 3. Phase 1 — Storage core  *(size: L)*

**Goal:** interface defined, both backends compile, doltlite can open/migrate an empty DB. No CLI commands beyond `version` yet.

### 3.1 `internal/store/` (interface + types + errors)

Create:
- `store.go` — `Store` interface, `Task`, `TaskPatch`, `ListFilter`, `Status` consts, `Rows`/`Result` minimal interfaces.
- `errors.go` — the sentinel errors listed in spec §6.1.
- `conformance/` — a shared test suite a backend can plug into:
  ```go
  func RunConformance(t *testing.T, newStore func(t *testing.T) Store)
  ```
  In Phase 1 this only covers `Open`, `Close`, `Migrate`, `SchemaVersion`. It grows in later phases.

### 3.2 `internal/migrations/`

- `001_init.sql` — exact SQL from spec §5.
- `migrations.go` — `//go:embed *.sql`, parses leading `NNN_` from filename, sorts numerically.
- `Apply(ctx, db *sql.DB) error` — runs each pending migration in its own transaction, records the row in `schema_migrations`. Idempotent.

### 3.3 `internal/projectdb/`

- `Discover(cwd string) (dbPath string, found bool, err error)` — walk up looking for `.autosk/db`.
- `AutoInit(cwd string) (dbPath string, err error)` — discover or create `<cwd>/.autosk/db`.
- Honor `AUTOSK_DB` env var as override.
- Honor `AUTOSK_NO_AUTOINIT=1` for tests that want to assert "no DB created".

### 3.4 `internal/store/doltlite/`

- `store.go` — `type Store struct { db *sql.DB }` implementing the `store.Store` interface (most methods return `ErrNotImplemented` for now).
- Implemented in this phase: `Open`, `Close`, `Migrate`, `SchemaVersion`.
- `Open` does:
  ```go
  db, _ := sql.Open("sqlite3", dbPath + "?_foreign_keys=on")
  db.SetMaxOpenConns(1)
  ```
- Compile-time interface assertion: `var _ store.Store = (*Store)(nil)`.
- `conformance_test.go` runs the shared suite against a `t.TempDir()` doltlite DB.

### 3.5 `internal/store/doltserver/`

- Build-tagged stub. File-level `//go:build doltserver`. Provides a stub `Store{}` that returns `errors.New("doltserver backend not implemented")` for every method.
- No tests; existence is the test.
- A separate file `doltserver_disabled.go` (without the tag) provides `var Available = false` so callers can branch on backend availability.

### Definition of done
- `make test` green; conformance suite (Phase-1 subset) passes for doltlite.
- `autosk version` reports `schema_version=1` after first run against a fresh DB.

## 4. Phase 2 — Create + show  *(size: M)*

**Goal:** scenario §11.1 turns green. A user can create and view a task.

### 4.1 ID generator (`internal/id/`)

- `New() string` returns `as-XXXX` where `XXXX` is 4 lowercase hex chars from `crypto/rand`.
- `NewUnique(ctx, exists func(id string) (bool, error)) (string, error)` — retries on collision (max 16 attempts).
- Prefix configurable via `Init()`; defaults to `as`. Loaded from a small `.autosk/config.json` written on init (single key for now).

### 4.2 doltlite impls

- `CreateTask(ctx, t Task) (Task, error)` — generates id if zero, sets timestamps, INSERT.
- `GetTask(ctx, id) (Task, error)` — SELECT by id, `ErrNotFound` on no row.

### 4.3 Commands

- `cmd/autosk/create.go`:
  - Positional title or `--title` (mutually exclusive with positional).
  - `-d, --description` string; `-d -` reads STDIN.
  - `-p, --priority N` (default 2).
  - `--blocks ID...` and `--blocked-by ID...` are **parsed but rejected** with a "not implemented until P4" error message. Cleaner than silently ignoring.
  - Auto-init via projectdb if no DB found.
  - On success: print id (human) or full JSON (`--json`).
- `cmd/autosk/show.go`:
  - Single positional id.
  - Pretty key/value block (human) or full JSON (`--json`). The `blocked` / `blocked_by` / `blocks` fields are present but always empty / `false` in this phase.

### 4.4 Renderer

- `internal/render/`:
  - `Task(w io.Writer, t Task)` — pretty key/value table.
  - `TaskJSON(w io.Writer, t Task)` — stable JSON shape (spec §10.2).
  - JSON keys frozen; documented in `internal/render/doc.go`.

### 4.5 Conformance suite additions

- Round-trip create → get.
- `CreateTask` with empty title → validation error.
- `CreateTask` with priority out of range → validation error.

### Definition of done
- `cd /tmp/x && autosk create "do the thing"` exits 0, prints `as-XXXX`, and produces `.autosk/db`.
- `autosk show as-XXXX` prints the task.
- `autosk show as-XXXX --json` round-trips through `jq` cleanly.
- Scenario §11.1 ✅.

## 5. Phase 3 — List / Ready / Claim / Update + status verbs  *(size: L)*

**Goal:** scenarios §11.2 and §11.4 turn green. Agents can loop.

### 5.1 doltlite impls

- `ListTasks(ctx, ListFilter)` — applied filter:
  ```sql
  SELECT ... FROM tasks
   WHERE (?statuses IS NULL OR status IN (...))
     AND (?priority IS NULL OR priority = ?)
   ORDER BY priority ASC, created_at ASC
   LIMIT ?
  ```
  Default `Statuses` when caller passes none: `{new, claimed}`. (Done/cancelled hidden by default.)
- `UpdateTask(ctx, id, TaskPatch)` — builds a partial UPDATE; sets `updated_at = now()`.
- `Claim(ctx, id)`:
  ```sql
  UPDATE tasks
     SET status = 'claimed', updated_at = ?
   WHERE id = ?
     AND status IN ('new','claimed')
  ```
  Inspect `RowsAffected`. If 0, run a fresh SELECT to distinguish `ErrNotFound` vs `ErrNotClaimable` (status is `done`/`cancelled`).
- `Ready(ctx, limit)` — full SQL (Phase-3 version, no edges yet, equivalent to "status='new' sorted by priority"):
  ```sql
  SELECT * FROM tasks WHERE status = 'new'
   ORDER BY priority ASC, created_at ASC
   LIMIT ?
  ```
  Phase 4 will replace this with the edge-aware version.

### 5.2 Commands

| File | Behavior |
|---|---|
| `cmd/autosk/list.go` | `--status`, `--priority`, `--limit`, `--json`. |
| `cmd/autosk/ready.go` | `--limit`, `--json`. |
| `cmd/autosk/next.go` | Calls `ready --limit 1`; exits 1 if nothing ready. |
| `cmd/autosk/claim.go` | Idempotent semantics. Prints the (post-state) task. |
| `cmd/autosk/done.go` | Sets status=done. |
| `cmd/autosk/cancel.go` | Sets status=cancelled. |
| `cmd/autosk/reopen.go` | Allows `done|cancelled → new`. Refuses `claimed → new` (use `update`). |
| `cmd/autosk/update.go` | Generic `--title/--description/--status/--priority`. |

### 5.3 Renderer

- `Tasks(w io.Writer, tasks []Task)` — table; columns `id`, `pri`, `status`, `title (truncated)`.
- `TasksJSON(w io.Writer, tasks []Task)` — JSON array.

### 5.4 Race-safety test (scenario §11.4)

A Go test launches N goroutines that each call `Claim(id)` on the same `new` task; exactly one row-affected=1 result is expected (the rest are 0 → idempotent no-op). The single-connection doltlite lock makes this trivially safe; the test still asserts it because future backends may not.

### Definition of done
- Scenarios §11.2 and §11.4 ✅.
- Conformance suite covers status transitions and `Claim` semantics.

## 6. Phase 4 — Blocking  *(size: L)*

**Goal:** scenario §11.3 turns green. Edges, cycle detection, derived blocked flag.

### 6.1 doltlite impls

- `Block(ctx, id, blockers...)`:
  ```go
  tx, _ := db.BeginTx(ctx, nil)
  // verify id exists
  // verify each blocker exists
  // reject if blocker == id  → ErrSelfBlock
  // reject if any blocker→...→id path exists → ErrCycle
  // INSERT OR IGNORE into task_deps for each pair
  tx.Commit()
  ```
- `Unblock(ctx, id, blockers...)` — DELETE matching rows in one tx.
- `UnblockAll(ctx, id)` — DELETE WHERE blocked_id=?, return RowsAffected.
- `Deps(ctx, id)` — two SELECTs (incoming + outgoing).
- `Ready` — replaced with the edge-aware version:
  ```sql
  SELECT t.* FROM tasks t
   WHERE t.status = 'new'
     AND NOT EXISTS (
       SELECT 1 FROM task_deps d
       JOIN tasks b ON b.id = d.blocker_id
       WHERE d.blocked_id = t.id
         AND b.status IN ('new','claimed')
     )
   ORDER BY t.priority ASC, t.created_at ASC
   LIMIT ?
  ```

### 6.2 Cycle detection

Naive iterative BFS from each proposed blocker, looking for `id` in the transitive `blocker_of(...)` set. Done inside the same tx that will insert the edges. Acceptable for MVP scale (hundreds of tasks). Document the O(V+E) per-edge cost as a known limitation.

### 6.3 Commands

- `cmd/autosk/block.go` — `autosk block <id> <blocker>...` (cobra `MinimumNArgs(2)`).
- `cmd/autosk/unblock.go` — accepts `<id> <blocker>...` **or** `<id> --all`. Cobra's `cobra.Args` custom validator enforces "ids OR --all but not both, and at least one of them".
- `cmd/autosk/dep.go` — `dep list <id>` only. Prints two sections (blocked_by, blocks).
- **Update `create.go`**: actually implement `--blocks` and `--blocked-by` now. They wrap `Block` calls inside the same logical operation (a separate tx per call since the task must exist first; document this).
- **Update `show.go`**: add derived `blocked` flag and the two arrays. SQL: one extra SELECT per show, or a `LEFT JOIN` aggregating ids.

### 6.4 Conformance additions

- Self-block rejection.
- Direct cycle (A blocks B, then `Block B A`) rejection.
- Transitive cycle (A→B→C, then `Block C A`) rejection.
- Cancelled blocker does not block (scenario §11.3 wording).
- `UnblockAll` removes only incoming edges.
- `Ready` correctness with mixed blocker statuses.

### Definition of done
- Scenario §11.3 ✅.
- `autosk show <id> --json` includes `blocked`, `blocked_by`, `blocks`.

## 7. Phase 5 — Polish + admin  *(size: M)*

**Goal:** scenarios §11.5 and §11.6 turn green. The product is feature-complete.

### Tasks
1. `cmd/autosk/init.go` — explicit init. Idempotent. Accepts `--prefix`. Errors if a `.autosk/db` exists in the cwd with a different prefix.
2. `cmd/autosk/migrate.go` — runs `store.Migrate`; prints how many migrations were applied (often 0).
3. `cmd/autosk/sql.go`:
   - Reads the query from a positional arg or `--file`.
   - Read-only by default: prepares with the doltlite read-only flag, or wraps in a savepoint and refuses non-SELECT statements.
   - `--write` opts in to writes (and produces an explicit doltlite commit afterwards using `SELECT dolt_commit(...)`).
   - Output: TSV by default; `--pretty` for table; `--json` for array of objects.
4. `cmd/autosk/history.go` — prints `"history delegates to doltlite log; planned for v0.2"`. Reserves the verb.
5. **JSON golden tests** under `cmd/autosk/testdata/`:
   - `show.golden.json`, `list.golden.json`, `ready.golden.json`.
   - Tests load goldens and `cmp.Diff` against runtime output.
   - Updateable via `go test -update`.
6. **Doltlite commit-on-write**: every write command, after a successful store mutation, runs `SELECT dolt_commit('-A','-m', '<verb> <id> [args]')`. Stub helper in `internal/store/doltlite/commit.go`. Read commands never commit.
7. **`README.md`** — install, quickstart, example agent loop, link to spec + plans.
8. **`AGENTS.md`** — pre-canned prompt block for AI agents (per beads convention): "use `autosk ready --json` to pick work, `autosk claim`, `autosk done`, never edit the DB directly".

### Definition of done
- All six acceptance scenarios ✅.
- `make test lint` green.
- README and AGENTS.md present.

## 9. Test strategy

- **Unit tests** live next to their files (`*_test.go`).
- **Conformance suite** in `internal/store/conformance/` — every backend must pass it. doltlite runs it now; doltserver runs it once implemented.
- **CLI acceptance tests** in `cmd/autosk/*_test.go` — spawn the binary with `t.TempDir()` and `AUTOSK_DB` set, assert stdout/exit code. Test data lives in `cmd/autosk/testdata/`.
- **Golden tests** for JSON output (Phase 5).
- **Race test** for `Claim` (Phase 3, scenario §11.4).
- **No production DB pollution**: every test uses `t.TempDir()`. The CLI refuses to run against a DB whose path contains `prod` unless `AUTOSK_ALLOW_PROD=1` (mirrors beads' guardrail).

## 10. Risk register

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| `libdoltlite` link is fragile on macOS (Python had to use `ctypes`) | Medium | High | Pin to a known-good doltlite commit in CI; `make doctor` catches missing artifacts; document `DOLTLITE_DIR`. |
| `mattn/go-sqlite3` + `libdoltlite` has surprising interactions (e.g., `PRAGMA foreign_keys=on` semantics) | Medium | Medium | Phase-0 smoke test exercises FKs; conformance covers cascade delete. |
| Cycle detection is slow at scale | Low (MVP) | Low | Document as a known limitation; revisit with recursive CTE if we ever need it. |
| Doltlite single-writer file lock blocks too aggressively | Low | Medium | `db.SetMaxOpenConns(1)` already serializes; document `AUTOSK_LOCK_TIMEOUT`. |
| Public module path / repo name not yet chosen | High | Low | Local-only `module autosk` works fine until first publish. |
